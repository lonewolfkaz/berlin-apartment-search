# Spec for JSON Import

branch: claude/feature/json-import

## Summary

Add a second input mode to the Add page: importing a listing from a `.json` file generated by Claude. The Add page gets a toggle/switcher between the existing manual form and a new JSON import view. The JSON view features a drag-and-drop zone that also supports manual file selection. A documented JSON API (schema) is provided so the user can share it with Claude to generate valid listing files.

## Functional Requirements

### JSON Schema / API

- Define and document a JSON schema that maps to the Listing data model
- Required fields: `address`, `district`, `price`, `size`, `rooms`, `floor`, `totalFloors`, `year`, `energy`
- Optional fields: `broker` (default `false`), `kfw`, `url`, `notes`, `ev` (evaluation object)
- The schema documentation should be a standalone reference the user can paste into a Claude conversation so Claude can generate valid `.json` files
- The schema should be stored as a constant in the codebase (e.g. `src/data/listingSchema.js`) and rendered in the UI as a copyable block

### Mode Switcher

- Add a toggle at the top of the Add page with two modes: **Manual** and **JSON Import**
- Default mode: Manual (preserves current behavior)
- Switching modes does not lose form state — if the user fills some manual fields, switches to JSON, then switches back, the manual form retains its values
- The switcher should follow the existing design system (DM Sans, pill-style toggle, color tokens from `T`)

### Drag-and-Drop Component

- A drop zone that accepts `.json` files
- Visual states: idle, drag-over (highlight), success (parsed), error (invalid)
- Also includes a "Browse files" button/link for manual file selection via native file picker
- Only accepts `.json` files (validate both by extension and by attempting `JSON.parse`)
- Single file at a time (not multi-file)

### Import Flow

- User drops or selects a `.json` file
- App reads the file, parses JSON, and validates against the listing schema
- On success: show a preview card of the parsed listing (reuse the existing preview style from the manual form — calculated all-in price, district, scenario dots)
- On validation error: show a clear, human-readable error message indicating which fields are missing or invalid, displayed inline in the drop zone area
- User confirms import by clicking a "Save listing" button (same CTA as manual mode)
- On save: listing is added via the same `onSave` callback used by the manual form
- `id`, `status` ("new"), and `created` are always generated by the app — never taken from the JSON file

### Schema Reference Panel

- Below the drop zone, show a collapsible section titled "JSON Schema for Claude"
- Contains the full JSON schema/example with field descriptions
- Includes a "Copy to clipboard" button so the user can paste it into a Claude chat

## Possible Edge Cases

- File is not valid JSON (syntax error) — show parse error message
- JSON is valid but missing required fields — show which fields are missing
- JSON has extra/unknown fields — ignore them silently, only use recognized fields
- `district` value doesn't match any entry in `DISTRICTS` — show validation error
- `energy` value doesn't match any entry in `ENERGY` — show validation error
- `price`, `size`, `rooms` etc. are strings instead of numbers — attempt coercion, fail if not numeric
- File has correct structure but `.txt` or other extension — reject, require `.json`
- User drops multiple files — only process the first one
- Very large file (e.g. someone drops a wrong file) — set a reasonable size limit (e.g. 100KB) and reject with a message

## Acceptance Criteria

- [ ] Add page has a visible toggle between Manual and JSON Import modes
- [ ] Manual mode works exactly as before (no regressions)
- [ ] Drag-and-drop zone accepts `.json` files and parses them
- [ ] "Browse files" button opens native file picker filtered to `.json`
- [ ] Valid JSON shows a listing preview with all-in price and scenario dots
- [ ] Invalid JSON or missing required fields shows clear inline error messages
- [ ] "Save listing" adds the imported listing to the app via existing `onSave` flow
- [ ] App always generates `id`, `status`, and `created` — ignores these from JSON
- [ ] Collapsible schema reference section with copy-to-clipboard is present
- [ ] Switcher preserves manual form state when toggling between modes
- [ ] Design follows existing tokens, fonts, and component patterns

## Open Questions

- Should the schema reference also be exported as a downloadable `.md` or `.txt` file, or is copy-to-clipboard sufficient? example file for another Claude agent can be saved in _specs folder as well
- Should the import support an array of listings (batch import) in the future, or strictly single-listing files? For now single listing is enough.

## Testing Guidelines

Create a test file(s) in the /tests folder for the new feature, and create meaningful tests for the following cases,
without going too heavy:

- JSON parsing: valid file produces correct listing object
- Validation: missing required fields returns appropriate errors
- Validation: unknown fields are silently ignored
- Validation: district and energy values are checked against known constants
- Field coercion: string numbers are converted, non-numeric strings are rejected
- Generated fields: `id`, `status`, `created` are always overwritten by the app
- File type rejection: non-`.json` files are rejected
- Mode switcher: toggling modes preserves manual form state
